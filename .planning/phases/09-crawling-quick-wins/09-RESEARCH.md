## Summary

This phase addresses foundational SEO and performance issues: robots.txt validation with sitemap reference, image dimension attributes for CLS prevention, hero image priority loading for LCP, and map embed layout stability. The requirements are straightforward HTML/CSS modifications with no complex dependencies.

The key constraint is that Cloudflare manages the robots.txt file by prepending AI crawl control directives. However, per Cloudflare documentation, they prepend their content BEFORE any existing robots.txt - meaning our sitemap directive will still be included and valid. The site already has a working `sitemap.xml` generated by Hugo at the root level.

Image dimension handling requires adding explicit `width` and `height` attributes to all `<img>` elements. The hero image additionally needs `fetchpriority="high"` to prioritize LCP loading. The map embed container already has fixed dimensions (400px height desktop, 350px mobile), but verification is needed.

**Primary recommendation:** Add sitemap directive to robots.txt template, add width/height to all images with actual pixel dimensions, add fetchpriority="high" to hero image, verify map container maintains fixed dimensions.

## Architecture Patterns

### robots.txt Template Pattern
**What:** Hugo can generate robots.txt from a template at `layouts/robots.txt`
**When to use:** When you need dynamic content or site-specific directives

```
layouts/
└── robots.txt        # Custom robots.txt template
```

### Image Dimension Pattern
**What:** Explicit width and height on all img elements using actual pixel values
**When to use:** Always - prevents Cumulative Layout Shift (CLS)

```html
<!-- Source: web.dev CLS optimization guide -->
<img src="image.jpg"
     width="1500"
     height="1000"
     alt="Description">
```

### Fetch Priority Pattern
**What:** Use fetchpriority="high" ONLY on the LCP image
**When to use:** Single above-fold hero image that is the Largest Contentful Paint element

```html
<!-- Source: MDN fetchpriority documentation -->
<img src="hero.jpg"
     fetchpriority="high"
     width="1500"
     height="1000"
     alt="Hero description">
```

### Embed Layout Reservation Pattern
**What:** Fixed dimensions or aspect-ratio on iframe containers to prevent CLS
**When to use:** Google Maps embeds, YouTube embeds, any third-party iframe

```css
/* Source: web.dev CLS optimization */
.embed-container {
  aspect-ratio: 16 / 9;  /* Modern approach */
  /* OR */
  height: 400px;         /* Fixed height approach (already in use) */
}
```

### Anti-Patterns to Avoid
- **Multiple fetchpriority="high" images:** Only one image should have high priority - using it on multiple images defeats the purpose
- **Omitting units in CSS but including them in HTML:** HTML width/height are unitless pixel values; CSS uses px units
- **Using loading="lazy" on hero image:** Conflicts with fetchpriority="high" and delays LCP

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| robots.txt generation | Manual file in static/ | Hugo template at layouts/robots.txt | Allows dynamic sitemap URL |
| Image dimension detection | Manual measurement | `file` or `sips` commands | Already have actual dimensions |
| Layout reservation for embeds | JavaScript measurement | CSS fixed height or aspect-ratio | CSS loads before JS |

**Key insight:** All solutions here are native HTML/CSS - no libraries or custom code needed.

## Common Pitfalls

### Pitfall 1: Cloudflare robots.txt Override
**What goes wrong:** Custom robots.txt content appears to be replaced by Cloudflare Content-Signal directives
**Why it happens:** Cloudflare prepends their managed content to existing robots.txt
**How to avoid:** Cloudflare prepends, not replaces - your sitemap directive will still appear after their block
**Warning signs:** robots.txt content only shows Cloudflare directives (means no source file exists)

### Pitfall 2: Wrong Dimension Values
**What goes wrong:** CSS overrides cause images to render at different sizes than specified
**Why it happens:** Using display dimensions instead of intrinsic (file) dimensions
**How to avoid:** Use actual file pixel dimensions; CSS max-width:100% still works correctly
**Warning signs:** Images stretch or compress unexpectedly

### Pitfall 3: fetchpriority on Below-Fold Images
**What goes wrong:** Hero image doesn't load faster despite fetchpriority="high"
**Why it happens:** Multiple images marked high priority, or LCP element isn't actually the hero
**How to avoid:** Only use fetchpriority="high" on ONE image - the LCP element
**Warning signs:** Network waterfall shows hero image loading after other resources

### Pitfall 4: loading="lazy" + fetchpriority="high" Conflict
**What goes wrong:** Hero image delayed despite high priority
**Why it happens:** lazy loading defers the request, contradicting high priority
**How to avoid:** Never combine loading="lazy" with fetchpriority="high"
**Warning signs:** Hero image loads late in waterfall despite priority hint

### Pitfall 5: Map Container Height Not Reserved
**What goes wrong:** Content shifts when iframe loads
**Why it happens:** Container has auto height that changes when iframe content loads
**How to avoid:** Use fixed height (already 400px) or CSS aspect-ratio
**Warning signs:** Visible content jump when scrolling to service area section

## Code Examples

### robots.txt Template with Sitemap
```
{{/* layouts/robots.txt */}}
{{/* Note: Cloudflare prepends AI crawl control; this content follows */}}
User-agent: *
Allow: /

Sitemap: {{ "sitemap.xml" | absURL }}
```

### Hero Image with fetchpriority
```html
<!-- layouts/index.html - hero section -->
<div class="hero-image">
    <img src="{{ "images/hero-image.jpg" | relURL }}"
         alt="Electrical work and installation"
         fetchpriority="high"
         width="1500"
         height="1000">
</div>
```

### Team Member Image with Dimensions

<!-- layouts/index.html - team section -->

     alt="Polos Electronics Logo"
